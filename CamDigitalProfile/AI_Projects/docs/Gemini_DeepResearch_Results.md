Architectural Implementation Guide for a Production-Grade Agentic Portfolio System Using Google Gemini and AWSThe integration of agentic artificial intelligence into a recruiter-facing portfolio requires a sophisticated balance of high-performance reasoning, cost-efficient infrastructure, and robust security. For an industry professional seeking to demonstrate technical excellence, the architecture must transcend a simple chat interface, evolving into a stateful, tool-augmented system capable of autonomous information retrieval and precise computation. The following analysis provides an exhaustive roadmap for implementing such a system using Google Gemini 1.5 as the primary reasoning engine, LangChain as the orchestration framework, and Amazon Web Services (AWS) as the hosting environment.1. Recommended ArchitectureThe architectural design for a production-quality AI agent on AWS is defined by the requirement for high availability, low latency, and a serverless cost profile. Given the bursty nature of recruiter traffic, the architecture must avoid persistent compute costs while ensuring that the "Time to First Token" (TTFB) remains minimal to maintain user engagement.Serverless Execution EnvironmentThe architecture MUST prioritize AWS Lambda for backend compute, specifically utilizing Lambda Function URLs to manage the execution of the LangChain agent. While Amazon API Gateway is a common choice for REST APIs, its 29-second timeout and lack of native support for HTTP chunked transfer encoding make it less suitable for streaming LLM responses. Lambda Function URLs, introduced as a direct endpoint for Lambda functions, support response streaming natively when wrapped with the awslambda.streamifyResponse utility in a Node.js runtime (version 18 or higher). This mechanism allows the agent to begin pushing tokens to the frontend as soon as they are generated by the Gemini model, rather than waiting for the entire response to complete.The execution environment SHOULD be configured with a minimum of 1760 MB of memory to ensure the underlying Lambda function is allocated a full vCPU, which is critical for the initialization of LangChain and its various dependencies. Tooling Best PracticesThe agent's utility is derived from its ability to interact with the external world. For a portfolio, the tools serve a dual purpose: they provide accurate information and demonstrate the candidate's ability to implement complex, side-effect-producing AI features.Secure Calculator ImplementationA common implementation error in AI agents is the use of the eval() function to perform calculations. This creates a critical security risk, as a malicious user could craft a prompt that tricks the agent into executing arbitrary system commands through the calculator tool. To mitigate this, the calculator tool MUST be implemented using a secure expression parser such as mathjs in Node.js or a restricted environment in Python.The mathjs library is the RECOMMENDED default for JavaScript-based portfolios because it supports a wide range of mathematical operations without ever calling the native eval(). It operates by parsing the input string into an expression tree and evaluating it within a controlled scope. The tool's description provided to Gemini SHOULD be explicit: "Use this tool for all mathematical operations, no matter how simple. Input MUST be a valid mathematical string." This discourages the LLM from attempting to perform math in its head, which often leads to hallucinations.Web Search Optimization with TavilyTavily is uniquely positioned as a search engine designed for LLMs, returning cleaned markdown or text snippets instead of raw HTML. This is critical for portfolio agents where token efficiency directly impacts cost and context window management. The implementation MUST use the TavilySearch tool with a max_results parameter set between 3 and 5.To improve the relevance of search results, the agent SHOULD be instructed via the system prompt to include the current date in its search queries. For example, a query like "Who won the Super Bowl?" is ambiguous, whereas "Who won the Super Bowl in 2025?" ensures the model retrieves the most recent information. The developer SHOULD also utilize Tavily's include_domains parameter to boost results from high-authority professional sites such as GitHub, LinkedIn, or the developer's own domain, ensuring the agent remains focused on the candidate's professional footprint.Tavily Search Parameter StrategyParameterRecommended SettingJustificationsearch_depthbasicControls costs; advanced is rarely needed for portfolio queries.max_results3Balance between information density and context window usage.include_answerfalseThe LLM will generate its own answer from the results; saves credits.include_raw_contentfalseRaw HTML is too token-heavy; use snippets or markdown.time_rangeyearEnsures the agent doesn't retrieve stale professional information.4. Security, Reliability, and Cost ControlsOperating a public AI agent on AWS introduces financial and security liabilities. A production-grade system must incorporate defensive layers to prevent abuse and ensure predictable billing.Prompt Injection MitigationPrompt injection—the act of using user input to override the agent's system instructions—is the primary threat to LLM applications. The developer MUST implement a "Defense in Depth" strategy. This begins with the use of XML delimiters in the system prompt (e.g., <system_instructions>...</system_instructions>) to help the model distinguish its core programming from the user's data.Furthermore, an input validator SHOULD be implemented as a lightweight middleware before the prompt reaches the LLM. This validator can use regex or a small, inexpensive model to detect keywords associated with injection, such as "ignore previous instructions" or "reveal system prompt". If an injection attempt is detected, the system should return a standard refusal message without invoking the more expensive Gemini 1.5 model.Rate Limiting and Quota ManagementTo prevent denial-of-service (DoS) attacks or automated scrapers from depleting the API budget, the architecture MUST include a rate-limiting layer. This is ideally implemented at the AWS WAF (Web Application Firewall) level or within the Lambda function using a simple DynamoDB-based token bucket algorithm. For a recruiter-facing site, a generous but firm limit—such as 10 queries per minute per IP address—is RECOMMENDED.Cost control SHOULD also be managed through Google AI Studio's billing limits. Additionally, the LangChain agent SHOULD be configured with max_iterations (e.g., 5-10) to prevent the model from entering a "reasoning loop" where it repeatedly calls tools without reaching a conclusion, which can occur if tool outputs are ambiguous or contradictory.Security and Reliability ChecklistCategoryRequirementRecommendation LevelImplementation DetailPrompt InjectionXML DelimitersMUSTWrap system prompt in <tags>.Code ExecutionNo eval()MUSTUse mathjs or sandboxed Python.Cost Controlmax_iterationsMUSTSet to < 10 to stop runaway loops.ReliabilityError HandlingSHOULDTools must return errors as strings to the LLM.MonitoringTracingSHOULDUse LangSmith for production observability.5. RAG + Memory (Phase 2+)As the portfolio agent matures, it must move beyond stateless interactions. The integration of long-term memory and document retrieval (RAG) allows the agent to provide personalized responses based on previous conversations and the developer's full project history.Transitioning to LangGraph for PersistenceFor Phase 2, the implementation MUST transition from standard LangChain chains to LangGraph. LangGraph introduces a stateful orchestration layer that treats the agentic loop as a graph of nodes and edges. The most critical advantage of LangGraph for production is its native support for "checkpointers".A checkpointer allows the agent's entire state—including the message history, internal reasoning steps, and tool outputs—to be persisted to a database. For an AWS-based stack, Amazon DynamoDB is the RECOMMENDED persistence backend. By using the langgraph-checkpoint-amazon-dynamodb library, the developer can assign a thread_id to each user session. When the recruiter returns or sends a new message, the agent retrieves its state from DynamoDB, allowing for a seamless, multi-session conversation.Retrieval-Augmented Generation (RAG) ArchitectureThe document retrieval system SHOULD utilize the Gemini 1.5 multimodal embedding models (text-embedding-004 or gemini-embedding-2-preview) to index the developer's resume, blog posts, and project documentation. These embeddings SHOULD be stored in a vector database. For a small-to-medium portfolio, pgvector on Amazon RDS is the most cost-effective and architecturally simple choice, as it allows the developer to store both relational data and vector embeddings in a single database.The RAG pipeline SHOULD implement a "hybrid search" strategy, combining semantic vector search with traditional keyword-based search (BM25). Testing and EvaluationEvaluation is the process of measuring the quality and reliability of an agent's outputs. For a recruiter-facing tool, failure to provide accurate information about the candidate can be more damaging than having no tool at all.Observability with LangSmithLangSmith MUST be enabled during the development and production phases. LangSmith provides "high-fidelity traces" of the agent's execution, allowing the developer to see the inputs and outputs of every internal step. This visibility is essential for identifying "tool selection errors"—cases where the model chooses the search tool when it should have used the calculator, or vice versa.Tracing is also critical for monitoring "hallucinations" in the RAG pipeline. By inspecting the retrieved document chunks in LangSmith, the developer can determine if the model is hallucinating information that was never present in the source documents.Evaluation Metrics for Agentic SystemsThe system SHOULD be evaluated against a "Golden Dataset" of common recruiter questions (e.g., "What technologies did you use for the AWS project?", "Can you calculate the cost-efficiency of your latest app?"). The evaluation SHOULD focus on three primary metrics:Tool Selection Correctness: A binary score (0 or 1) indicating if the model invoked the appropriate tool for the query.Trajectory Accuracy: Measuring if the agent took the expected path to an answer. For a multi-step query, did it perform the sub-tasks in a logical order?.Faithfulness (for RAG): Ensuring that the final answer is strictly grounded in the retrieved context and does not include external training data knowledge that contradicts the developer's specific project history.7. Deliverables for Immediate UseTo facilitate immediate implementation within an environment like Cursor, the following code structures and configuration templates are provided.Secure Calculator Tool Template (TypeScript)This implementation uses mathjs to provide a safe, non-evaluating calculator that can be registered as a LangChain tool.TypeScriptimport { tool } from "@langchain/core/tools";
import { create, all } from "mathjs";
import { z } from "zod";

/**
 * A secure calculator implementation for LangChain agents.
 * This avoids the use of eval() by using the mathjs expression parser.
 */
const math = create(all);

export const secureCalculator = tool(
  async ({ expression }) => {
    try {
      // Evaluation is performed in a restricted scope provided by mathjs
      const result = math.evaluate(expression);
      return String(result);
    } catch (error) {
      // Returning errors as strings allows the LLM to 'reason' about the failure
      return `Error: Invalid mathematical expression. ${error instanceof Error? error.message : ""}`;
    }
  },
  {
    name: "calculator",
    description: "Evaluates mathematical expressions. Mandatory for any math or numerical task.",
    schema: z.object({
      expression: z.string().describe("The math expression to solve, e.g., 'sqrt(144) + 12'"),
    }),
  }
);

AWS Lambda Streaming Handler (Node.js)This handler demonstrates the correct way to implement response streaming using Lambda Function URLs and LangChain.JavaScriptimport { ChatGoogleGenerativeAI } from "@langchain/google-genai";
import { StringOutputParser } from "@langchain/core/output_parsers";

// Wrap the handler with streamifyResponse for Function URL streaming
export const handler = awslambda.streamifyResponse(
  async (event, responseStream, context) => {
    const model = new ChatGoogleGenerativeAI({
      apiKey: process.env.GOOGLE_API_KEY,
      modelName: "gemini-1.5-flash",
      streaming: true,
    });

    const body = JSON.parse(event.body);
    const userMessage = body.message;

    // Set headers for streaming response
    const metadata = {
      statusCode: 200,
      headers: { "Content-Type": "text/plain" },
    };
    
    responseStream = awslambda.HttpResponseStream.from(responseStream, metadata);

    const stream = await model.stream(userMessage);

    for await (const chunk of stream) {
      // Write each token directly to the stream
      responseStream.write(chunk.content);
    }

    responseStream.end();
  }
);
